# `.from()` & Factory Ergonomics — Design

## Problem

Five gaps between the current `.from()` / factory API and the ergonomic target:

1. No-arg factory calls fail for nodes with only optional fields
2. Omitted required wrapper fields don't default to empty
3. Arrays at wrapper positions don't auto-wrap
4. Single values at wrapper positions don't auto-wrap
5. Single-field factories force an unnecessary config object

## Design

### Gap 1: Optional config on all-optional-field factories

**Detection:** `resolveConfigOptional` already returns `'?'` when no required non-auto-stamp fields exist. Currently this only affects the `?` on the param — the factory still expects a defined value at runtime because `config.children ?? []` needs `config` to exist.

**Fix:** When `resolveConfigOptional` returns `'?'`, emit:
```ts
export function block(config?: T.Block.Config) {
  const _config = config ?? {};
  const children = _config.children ?? [];
  const _label = _config.label;
```

The `?? {}` supplies a default empty config. All field reads use `_config` (which is always defined).

**Scope:** Any factory where `resolveConfigOptional` returns `'?'` — already computed, just underutilized.

### Gap 2: Default empty for required container fields in `.from()`

**Detection at codegen time:** A field is eligible for default-empty when:
- It is required (`isRequired(f)`)
- Its values resolve to exactly ONE kind
- That kind's factory has a no-arg signature (per Gap 1)

**Emit:** `_resolveOneBranch<T.Block>(input.body, "block") ?? F.block()`

The `?? F.block()` fallback fires when the input omits the field entirely (`undefined`).

### Gap 3: Array → wrap with children

**Detection at resolve time:** `Array.isArray(v)` in `_resolveOneBranch`.

**Emit a new branch in the generated `_resolveOneBranch` helper:**
```ts
if (Array.isArray(v)) {
    const resolved = v.map(e => _resolveChildFor(e, kind));
    return _wrapWithChildren(kind, resolved) as T;
}
```

**`_wrapWithChildren`:** Calls the target kind's factory with `{ children: resolved }`. Generated per-grammar as a dispatch table keyed by kind string → factory call.

**Child-type metadata:** Per-kind table emitted by codegen: `{ kind: string, childKinds: string[] }`. Used by `_resolveChildFor` to route each element through the correct resolver.

### Gap 4: Single value → wrap with children

**Detection at resolve time:** Value is NodeData with `$type !== targetKindId`, or value is a plain object that doesn't match the target kind.

**Emit in `_resolveOneBranch`:**
```ts
if (isNodeData(v) && v.$type !== kindId) {
    return _wrapWithChildren(kind, [v]) as T;
}
if (typeof v === 'object' && v !== null && !Array.isArray(v)) {
    const resolved = _resolveChildFor(v, kind);
    return _wrapWithChildren(kind, [resolved]) as T;
}
```

The `$type` check is the discriminator: if the value IS the wrapper kind, pass through. Otherwise, wrap as a single child.

### Gap 5: Single-field factory signatures

**Detection at codegen time:** `fields.length === 1 && children.length === 0 && !autoStamp`

**Emit:** Direct-value signature instead of config-object:
```ts
export function typeAnnotation(type: T.TypeAnnotationType) {
    const _type = type;
    return withMethods({ ... });
}
```

**`.from()` resolution:** The from-resolver for single-field kinds accepts BOTH:
- Direct value: `ir.typeAnnotation.from(someType)` — the value IS the field
- Config bag: `ir.typeAnnotation.from({ type: someType })` — existing path

Detection: if `typeof input === 'object' && '$type' in input` → NodeData passthrough. If `typeof input === 'object' && !('$type' in input)` → config bag. Otherwise → direct value for the single field.

### Shared infrastructure

**`_wrapWithChildren(kind, children)`** — dispatch table:
```ts
function _wrapWithChildren(kind: string, children: unknown[]): unknown {
    switch (kind) {
        case 'block': return F.block({ children });
        case 'parameters': return F.parameters(...children);
        // ... per-kind entries for wrapper-shaped nodes
    }
}
```

Generated by codegen. Only includes kinds that have `$children` (container-shaped or mixed-shape with children).

**`_resolveChildFor(value, parentKind)`** — routes a child value through the correct resolver based on the parent kind's declared child types. Uses the per-kind child-type metadata table.

## Implementation order

1. Gap 1 first (prerequisite for Gap 2)
2. Gap 5 (independent, no dependencies)
3. Gap 2 (depends on Gap 1)
4. Gaps 3+4 together (share `_wrapWithChildren` infrastructure)

## Success criteria

- `ir.block()` compiles and produces an empty block
- `ir.functionItem.from({ name: 'main' })` produces a function with empty body + parameters
- `ir.parameters.from([p1, p2])` wraps both in a parameters node
- `ir.block.from(stmt)` wraps a statement in a block
- `ir.typeAnnotation(someType)` takes the value directly (no config object)
- All existing fromPass/factoryPass counts hold or improve
- Zero type-check errors
