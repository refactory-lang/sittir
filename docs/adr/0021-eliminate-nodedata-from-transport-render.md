# ADR 0021 — Eliminate NodeData from Transport Render Path

**Status**: Proposed
**Date**: 2026-05-06
**Related**: ADR 0019, ADR 0020, specs/024-streaming-render/

## Context

The transport render path currently builds a full `NodeData` from
`AnyTransport` solely to extract two pieces of metadata:

1. **Trivia** — `node.trivia_data` (leading/trailing comment text)
2. **Format** — `node.type_` + `node.span` (for `render_canonical_node`
   which applies indentation/whitespace formatting)

This is wasteful: `node_data_from_transport` constructs a `HashMap` of
fields, `Vec` of children, clones strings — all to read two metadata
properties that are already available on the transport struct itself.

The transport path should render WITHOUT constructing `NodeData` at all.

## Current flow (wasteful)

```
JS factory node with $trivia + $format
  → napi FromNapiValue → AnyTransport
  → render_transport_parts():
      1. render_transport_dispatch(&transport) → String (streaming, correct)
      2. node_data_from_transport(transport) → NodeData (WASTEFUL)
      3. read node.trivia_data for wrapping (could read from transport directly)
      4. return (NodeData, rendered) to caller
  → caller: engine.render_canonical_node(&node, canonical, tree_format)
      - reads node.type_ for format lookup
      - reads node.span for boundary detection
```

## Decision

Eliminate `NodeData` construction from the transport render path entirely.
Both trivia and format metadata should be read directly from the transport
struct or passed as separate lightweight values.

### Phase 1: Trivia without NodeData

Trivia is `TransportTrivia { leading: Option<Vec<String>>, trailing: Option<Vec<String>> }`.
It's already on each transport struct as `transport_trivia_data`.

**Fix**: The napi entry point (`SittirEngine::render` in `lib.rs`) reads
`$triviaData` from the JS object BEFORE constructing `AnyTransport`. It
passes the `Option<TransportTrivia>` as a separate value to the render
function. Trivia wrapping happens after `render_transport_dispatch`
returns the rendered string — no NodeData involved.

```rust
// In lib.rs napi render method:
pub fn render(&mut self, transport: AnyTransport, trivia: Option<TransportTrivia>) -> Result<String> {
    let mut rendered = render_transport_dispatch(&transport)?;
    // Apply trivia (just string prepend/append)
    if let Some(t) = &trivia {
        // ... push_str leading/trailing ...
    }
    // Apply format (see Phase 2)
    Ok(rendered)
}
```

### Phase 2: Format without NodeData

`render_canonical_node` needs:
- `node.type_` → `KindId` (for format rule lookup)
- `node.span` → `Option<Span>` (for boundary detection)

Both are available on the transport struct metadata fields:
- `transport_source` → already present
- The `KindId` is known at dispatch time (the match arm constant)

**Fix**: Change the format application API from
`render_canonical_node(&NodeData, String, Option<Format>)` to
`apply_render_format(kind_id: KindId, span: Option<Span>, canonical: String, format: Option<Format>) → String`.

The caller passes the KindId (known from the transport dispatch) and
span (from `transport_span`) directly. No NodeData construction.

```rust
// In lib.rs:
pub fn render(&mut self, transport: AnyTransport, trivia: Option<TransportTrivia>) -> Result<String> {
    let kind_id = transport.kind_id();  // method on AnyTransport
    let span = transport.span();        // reads transport_span
    let mut rendered = render_transport_dispatch(&transport)?;
    // Trivia
    apply_trivia(&mut rendered, &trivia);
    // Format
    let tree_format = self.parsed.as_ref().and_then(|pt| pt.format());
    rendered = apply_render_format(kind_id, span, rendered, tree_format);
    Ok(rendered)
}
```

### Phase 3: Remove node_data_from_transport

Once Phases 1+2 land, `node_data_from_transport` has no callers in
the render path. It may still be needed by:
- The `readNode` napi path (if it constructs NodeData from transport)
- Test fixtures

Audit callers and remove if dead. If still needed for non-render
purposes (e.g., structural inspection), keep but move out of the
render hot path.

### Required helpers on AnyTransport

The enum needs two accessor methods:

```rust
impl AnyTransport {
    /// Extract the KindId — compile-time constant per variant.
    fn kind_id(&self) -> KindId { ... }  // 700-arm match

    /// Extract the span metadata (if present).
    fn span(&self) -> Option<Span> { ... }  // reads transport_span
}
```

These are generated by the emitter (one match arm per variant).
Alternatively, the napi entry point reads `$type` and `$span` from
the JS object BEFORE constructing `AnyTransport` — avoiding the match.

## Consequences

- **Zero `NodeData` allocation in the transport render path** — no HashMap,
  no Vec, no string cloning for fields/children
- **Trivia wrapping is just string prepend/append** — `Vec<String>` texts
  written directly
- **Format application is two scalars** — `KindId` + `Option<Span>`, not
  a full NodeData struct
- **`node_data_from_transport` becomes dead code** on the render path
- **Follow-up**: if `render_dispatch` (NodeData path) is also eliminated
  per ADR 0020 follow-up A, then `NodeData` is only constructed by
  `readNode` — never in the render path at all

## Alternatives considered

- **Keep NodeData but make it cheaper** — lazy fields, skip HashMap.
  Still more expensive than not constructing it at all.
- **Extract metadata into a separate `TransportMeta` struct** — cleaner
  than NodeData but still an intermediate allocation. Direct scalar reads
  are simpler.
- **Move trivia wrapping to JS side entirely** — possible for factory
  nodes, but doesn't work for readNode-derived nodes rendered natively.
  Both paths need it.
