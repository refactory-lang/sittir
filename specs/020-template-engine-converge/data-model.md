# Data Model: Render Pipeline Optimization (020)

## Overview

This feature does not add a new persisted data store. Its "data model" is the set of runtime/codegen entities that change shape or lifecycle as the native render pipeline moves from the retained convergence baseline through Level 1 and Level 3.

---

## 1. Canonical Template Bundle

**Purpose**: The one authoritative per-grammar `.jinja` template set shared by both backends.

| Field | Type | Notes |
| --- | --- | --- |
| `grammar` | enum (`rust` \| `typescript` \| `python`) | Supported grammar package |
| `templateDirectory` | repo path | `packages/{lang}/templates/` |
| `templatesByKind` | map(kind -> template path) | One template per kind |
| `sourceOfTruth` | boolean | Always true for this bundle |

**Relationships**
- Read by the JS renderer through the existing template path.
- Read indirectly by the native render modules through generated Askama bindings.
- Must stay singular even while Level 1 and Level 3 change native population logic.

---

## 2. Grammar-Owned Native/Render Crate

**Purpose**: The grammar-specific native crate that consumes canonical templates, owns generated render code, loads the tree-sitter binding, and exposes native rendering for one grammar.

| Field | Type | Notes |
| --- | --- | --- |
| `grammar` | enum | Matches one canonical template bundle |
| `cratePath` | repo path | `rust/crates/sittir-{lang}/` |
| `crateName` | string | `sittir-{lang}` |
| `generatedSources` | set(path) | Includes `src/render/templates.rs`, `src/render/hash.rs`, `src/render/mod.rs`, `templates/*.jinja`, and helpers emitted by codegen |
| `levelState` | enum (`baseline`, `level1`, `level3`) | Current optimization stage for the crate during development |

**Relationships**
- Generated by `packages/codegen/src/emitters/render-module.ts` and related codegen flow.
- Depends on shared runtime primitives from `rust/crates/sittir-core`.
- One crate exists per supported grammar; a level is complete only when all three crates reach that level.

---

## 3. TemplateContext (Legacy Runtime Transport)

**Purpose**: The current native preparation-layer structure that carries rendered field and child values into generated Askama views.

| Field | Type | Notes |
| --- | --- | --- |
| `fields` | map(field -> string) | Scalar rendered values |
| `fields_list` | map(field -> list<string>) | Repeated rendered values |
| `children` | string | Joined rendered children |
| `children_list` | list<string> | Individual rendered children |
| `variant` | string | Variant selector when present |
| `text` | string | Raw/leaf text fallback |
| `leading_sep` / `trailing_sep` | boolean | Flank metadata used by join helpers |
| `lifecycle` | enum (`active`, `borrow-source-only`, `retired`) | Active in baseline, borrow-source-only in Level 1, retired after Level 3 cleanup |

**Relationships**
- Produced by the legacy preparation layer in `sittir-core`.
- Consumed directly by generated Askama structs in baseline and Level 1.
- Removed from the main runtime path after Level 3 is proven.

---

## 4. Borrowed Askama View

**Purpose**: The Level 1 generated struct surface that projects `TemplateContext` into Askama without cloning.

| Field | Type | Notes |
| --- | --- | --- |
| `kind` | string | Template-backed node kind |
| `scalarFields` | borrowed string refs | `&str`-style borrowed access from `TemplateContext.fields` |
| `listFields` | borrowed list refs | Borrowed slices for repeated values |
| `children` | borrowed list/string refs | Borrowed child view over `children_list` / joined output |
| `variant` / `text` | borrowed string refs | No owned clone boundary |
| `sourceTransport` | reference | Borrowed from one `TemplateContext` instance |

**Validation rules**
- Must not require authored template changes.
- Must not copy template-backed scalar/list values out of the source transport.
- Must preserve the same observable Askama field names and template behavior.

**Relationships**
- Exists only in Level 1 and any transitional code shared with early Level 3.
- Replaced or subsumed by direct render functions once Level 3 finishes.

---

## 5. Direct Render Function

**Purpose**: The Level 3 generated per-kind render entry that resolves field values directly from `NodeData` and constructs the Askama struct without `TemplateContext`.

| Field | Type | Notes |
| --- | --- | --- |
| `kind` | string | Template-backed node kind |
| `fieldStrategies` | map(field -> strategy) | Strategy is one of `leaf`, `optional-leaf`, `required`, `optional`, `list`, `children`, `variant` |
| `metadataBindings` | map(name -> emitted constant/routing rule) | Includes separators, leaf knowledge, join behavior, variant routing |
| `stackOwnedRenders` | list<string binding> | Owned strings for recursively rendered structured fields |
| `borrowedLeafs` | list<string binding> | Borrowed text slices from `NodeData` |
| `fallbackBehavior` | enum (`leaf-text`, `template-render`, `empty`) | Defines behavior for non-structured / non-template cases |

**Validation rules**
- Must preserve leaf short-circuiting, child ordering, separator/flank behavior, variants, and current format-aware rendering.
- Must be emitted from model-derived facts, not handwritten per-grammar logic.
- Cleanup of the legacy preparation layer is forbidden until parity proves the emitted function equivalent.

**Relationships**
- Uses the Resolve Helper Set.
- Replaces the `TemplateContext` path for template-backed kinds at Level 3.

---

## 6. Resolve Helper Set

**Purpose**: Shared helper surface used by Level 3 render functions to read from `NodeData` consistently.

| Helper Family | Responsibility |
| --- | --- |
| `resolve_leaf` / `resolve_optional_leaf` | Borrow leaf text directly from nested `NodeData` or text values |
| `resolve_required` / `resolve_optional` | Recurse for structured fields only when needed |
| `resolve_list` | Render repeated fields and preserve join semantics |
| `resolve_children` | Walk/render children plus separator/flank behavior |
| Variant helpers | Pick the correct child/field routing for variant-sensitive kinds |

**Placement**
- Shared generic logic belongs in emitted/native-render helper code, not in a new handwritten per-grammar abstraction.
- Any surviving shared runtime primitive must justify living in `sittir-core`; otherwise it stays with the generated render modules.

---

## Lifecycle Summary

| Entity | Baseline | Level 1 | Level 3 |
| --- | --- | --- | --- |
| Canonical Template Bundle | Active | Active | Active |
| Grammar-Owned Native/Render Crate | Active | Active | Active |
| TemplateContext | Active transport | Borrow source only | Retired from runtime path |
| Borrowed Askama View | Absent | Active | Transitional only |
| Direct Render Function | Absent | Absent | Active |
| Resolve Helper Set | Absent | Absent | Active |
