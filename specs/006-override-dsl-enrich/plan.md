# Implementation Plan: Override DSL with enrich(base) and Tree-Sitter-Compatible Authoring

**Branch**: `006-override-dsl-enrich` | **Date**: 2026-04-15 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/006-override-dsl-enrich/spec.md`

## Summary

Extend sittir's override system so `packages/<lang>/overrides.ts` is simultaneously runnable by tree-sitter (as a grammar extension file) and by sittir's codegen pipeline. Three pillars: (1) a pure `enrich(base)` function applied as `grammar(enrich(base), {...})` that performs mechanical-only auto-transformations (keyword-prefix promotion, unambiguous kind-to-name wrapping, repeat normalization) with skip reporting to stderr; (2) effect-accumulator DSL primitives (`role()`) scoped per grammar invocation; (3) transform API enhancements (path addressing, wildcards, nested patches, `alias($.name)` shorthand) run in a pre-evaluate phase. Override files import sittir DSL primitives explicitly from `@sittir/codegen` (dual CJS+ESM build) and are transpiled to `.sittir/grammar.js` for tree-sitter consumption. Top-level grammar arrays (`supertypes`, `externals`, `extras`) merge with spread+dedupe; `word` replaces.

Existing mechanical auto-inference passes in the Link phase move into `enrich`. Rust, TypeScript, and Python must continue to pass their fidelity ceilings.

## Technical Context

**Language/Version**: TypeScript (ESM, `.ts` extensions in imports), TypeScript 6.0.2
**Primary Dependencies**: `@sittir/core`, `@sittir/types`, `@sittir/codegen`; tree-sitter grammar packages (`tree-sitter-rust`, `tree-sitter-typescript`, `tree-sitter-python`); tree-sitter CLI for CI validation
**Storage**: File system — `packages/<lang>/overrides.ts` (hand-authored source), `packages/<lang>/.sittir/grammar.js` (transpiled output, gitignored), generated `.ts`/`.yaml` artifacts
**Testing**: Vitest unit tests for `enrich()` passes and transform API; existing round-trip and factory-round-trip validators for integration; new CI step running `tree-sitter generate` against `.sittir/grammar.js`
**Target Platform**: Node.js (ESM runtime for sittir pipeline; CJS runtime for tree-sitter CLI)
**Project Type**: compiler (five-phase codegen pipeline consuming grammar.js + tree-sitter metadata, emitting TypeScript)
**Performance Goals**: Full codegen run for all three grammars completes in under current baseline (no regressions). `enrich()` passes run in O(rules × pass-count) with no quadratic blowup.
**Constraints**: Zero runtime dependencies in generated packages (constitution). `enrich(base)` must be pure. Side-effect accumulators must use save/restore scoping — no leakage across nested `grammar()` calls. Fidelity ceilings (rust 50/40, typescript 10/10, python 40/30) must not regress.
**Scale/Scope**: Three currently-supported grammars (rust, typescript, python). ~500 kinds across all three. Override files typically 50-200 transformed rules each.

| **III. Generated vs Hand-Written** | `packages/<lang>/overrides.ts` is hand-written. `packages/<lang>/.sittir/grammar.js` is generated by the transpile step and gitignored. Clear separation preserved. | PASS |
| **IV. Test-First** | `enrich()` passes unit-tested against grammar fixtures before porting from Link. Transform API tested with before/after grammar snapshots. Integration gate: existing round-trip + factory-round-trip ceilings must not regress (FR-021). | PASS |
| **V. Library-First** | No CLI changes. Transpile step runs as part of the existing codegen pipeline. Tree-sitter CLI invocation happens in CI, not inside sittir. | PASS |

## Phase 0 — Outline & Research

See [research.md](research.md).

**Research topics**:

1. **Link-phase passes to port**: enumerate every auto-inference pass currently inside `packages/codegen/src/compiler/link.ts` and classify each as mechanical (moves to `enrich`) or heuristic (stays). Concrete list required because FR-020 is vague about which passes qualify.
2. **tree-sitter CLI invocation contract**: what does `tree-sitter generate` expect at `.sittir/grammar.js`? `tree-sitter.json` schema (where is it looked up from)? exit codes? Needed for FR-022 CI step.
3. **TS→JS transpile strategy**: `tsc`? `esbuild`? `swc`? Constraints: must preserve `@ts-nocheck`, must produce CJS output tree-sitter can `require()`, must resolve imports from `@sittir/codegen/dsl` after transpile.
4. **Accumulator save/restore pattern**: how does this work when the same `overrides.ts` module is imported multiple times (module caching) vs re-evaluated? Risk: stale state between codegen runs.
5. **Path addressing syntax for transforms**: `'0/0/0'` is the proposed shape. Confirm delimiter choice (slash vs dot), escape rules, wildcard character (`*`?), and whether leading/trailing slashes are significant.
6. **Extension-point dedupe identity**: for `externals`/`extras`/`supertypes`, what counts as "the same entry"? Reference equality? Value equality on the symbol name? Both `$.foo` and `$.foo` from different `$` proxies — are they equal?

## Phase 1 — Design & Contracts

See [data-model.md](data-model.md), [contracts/dsl-surface.md](contracts/dsl-surface.md), [quickstart.md](quickstart.md).

**Entities** (details in data-model.md):
- Base Grammar, Enriched Grammar, Override File, Role Registry, Transform Patch, Post-Transform Grammar, Skip Report

**Contracts** (details in contracts/dsl-surface.md):
- `enrich(base: Grammar): Grammar` — pure, mechanical-only
- `role(name: string): Blank` — side-effect + neutral return
- `transform(rule, patch | ...patches): Rule` — path-addressed + wildcards + nested
- `insert(rule, path, value): Rule` / `replace(rule, path, value): Rule`
- `field(name, content)`, `alias(target[, name])` — tree-sitter DSL re-exports with sittir shorthands
- Extension-point merge: `mergeExtensions(base, overrides): { supertypes, externals, extras, word }`

**Quickstart**: minimal `overrides.ts` example showing `grammar(enrich(base), { externals: [...base.externals, role('indent')], rules: { ... } })` — the canonical pattern a maintainer should start from.

**Agent context update**: run `.specify/scripts/bash/update-agent-context.sh claude` at the end of Phase 1 to record new technology (tree-sitter CLI in CI, dual CJS+ESM build of `@sittir/codegen`).
