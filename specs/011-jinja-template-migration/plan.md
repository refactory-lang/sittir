# Implementation Plan: Jinja Template Migration (Askama-Style)

**Branch**: `011-jinja-template-migration` | **Date**: 2026-04-21 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/011-jinja-template-migration/spec.md`

## Summary

Retire the ast-grep-flavored YAML template surface (`packages/{rust,typescript,python}/templates.yaml`) in favor of per-rule `.jinja` files stored under `packages/<grammar>/templates/`. Phase A swaps the TypeScript render engine's hand-rolled regex substitutor for Nunjucks, emitting one `.jinja` file per rule via the codegen template emitter. Phase B ships Rust-side askama structs (generated by codegen) that compile the same `.jinja` files at `cargo build`, with a cross-render parity test guaranteeing byte-identical output.

The feature rests directly on ADR-0013's completed cleanup: task 1 (loader split — render layer is browser-safe), task 2 (variant collapse — template surface shrank from 21 variant blocks to 5), and task 3 (`prepare()` — produces the `TemplateContext` shape templates consume). All three are on `origin/master` (`7db681d`, `17d4749`, `a047294`).

## Technical Context

**Language/Version**: TypeScript 6.0.2 (ESM, `.ts` extensions in imports). Rust edition 2024 for Phase B (port target).
**Primary Dependencies**: `@sittir/core`, `@sittir/types`, `@sittir/codegen` (workspace). Adds `nunjucks` (TS render) in Phase A. Adds `askama` crate in Phase B.
**Storage**: File system — `.jinja` files under `packages/<grammar>/templates/<rule_kind>.jinja`. No YAML wrapper. `factory-map.json5`, `node-model.json5`, and `overrides.ts` unchanged.
**Testing**: Vitest for TS (existing corpus + round-trip + new translator-parity unit tests). Phase B adds `cargo test` cross-render parity tests in a new crate.
**Target Platform**: Node.js 20+ for TS codegen + test runtime; browser-safe for `@sittir/core/render` (per ADR-0013 Task 1). Rust 2024 toolchain for Phase B.
**Project Type**: Compiler / codegen library. Monorepo: `@sittir/core` (runtime), `@sittir/types` (pure types), `@sittir/codegen` (codegen), `@sittir/{rust,typescript,python}` (generated packages).
**Performance Goals**: Phase A — `pnpm test` wall-clock does not regress by more than 10% vs. the regex-substitutor baseline (Nunjucks has precompilation; template bodies are short). Phase B — Rust askama render beats TS Nunjucks render by at least 2× per node on the corpus (no runtime template parsing).
**Constraints**: Byte-identical rendering on the full round-trip corpus per phase. No template constructs outside the Nunjucks ∩ askama intersection. Compile-time template validation on the Rust side (unknown variables → build errors).
**Scale/Scope**: ~160 rules/grammar × 3 grammars ≈ 480 `.jinja` files. Existing corpus is ≈200 nodes per grammar. Phase A ships entirely in the `@sittir/*` workspace; Phase B adds one new Rust crate.

**No violations requiring Complexity Tracking entries.**

One implicit risk worth noting: adding `nunjucks` as a dependency to `@sittir/core` means one more JS runtime library the Rust port cannot reuse. Mitigation: the port replaces Nunjucks with askama — they consume the same `.jinja` assets. `@sittir/core` is not ported wholesale; Rust reimplements the render primitive using askama. This is consistent with Principle IX's "reimplementation obligation" framing.

**Structure Decision**: Phase A lands entirely inside the existing `@sittir/*` TypeScript workspace — no new packages or crates. Templates move from `packages/<grammar>/templates.yaml` to `packages/<grammar>/templates/*.jinja`. The codegen template emitter (`packages/codegen/src/emitters/templates.ts`) replaces its YAML-dictionary output with a multi-file `.jinja` emission. Phase B adds a new Rust crate `rust/crates/sittir-{lang}/src/render/` (first Rust code in the repo) consuming the same `.jinja` files via askama's `#[derive(Template)]`. Directory layout chosen so the Phase B crate reads `.jinja` files via relative paths from the generated grammar packages; askama resolves them at build time.
