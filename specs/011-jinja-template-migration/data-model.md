# Phase 1 Data Model: Jinja Template Migration

**Feature**: 011-jinja-template-migration
**Date**: 2026-04-21

This feature introduces no new persistent entities. It reshapes two existing
surfaces: the `TemplateContext` consumed by the render pipeline, and the
on-disk template asset layout. Both are formalized here.

---

## 1. `TemplateContext`

The input to every template render. Identical shape on TS (object) and Rust (struct).

### TypeScript declaration

Lives in `packages/core/src/render.ts` (post-ADR-0013 Task 3 already declares `PreparedRender`; this feature extends it to the template-engine-facing shape).

```ts
export interface TemplateContext {
	/**
	 * Pre-rendered named field slots keyed by raw (snake_case) field name.
	 * `undefined` when the field is absent on the node; optional fields may
	 * legitimately carry `undefined`.
	 *
	 * Field names come from the tree-sitter grammar verbatim. No camelCase
	 * translation — templates see the raw names.
	 */
	readonly [fieldName: string]: string | undefined | readonly string[] | boolean;

	/** Pre-joined unconsumed named children (with joinBy / flankSep applied). */
	readonly children: string;

	/** Individual pre-rendered children, in source order. Used by `{% for %}`. */
	readonly children_list: readonly string[];

	/** Node's $variant field, or empty string. */
	readonly variant: string;

	/** Node's $text field, or empty string. */
	readonly text: string;

	/** Whether a trailing separator was present adjacent to the named children. */
	readonly trailing_sep: boolean;

	/** Whether a leading separator was present adjacent to the named children. */
	readonly leading_sep: boolean;
}
```

### Rust declaration (Phase B)

One struct per rule, auto-generated by codegen. Each struct embeds the shared fields plus a typed projection of the rule's declared fields:

```rust
use askama::Template;

#[derive(Template)]
#[template(path = "rust/function_item.jinja")]
pub struct FunctionItemContext<'a> {
    // Shared slots
    pub children: &'a str,
    pub children_list: &'a [String],
    pub variant: &'a str,
    pub text: &'a str,
    pub trailing_sep: bool,
    pub leading_sep: bool,

    // Per-rule named field slots (generated from AssembledField list)
    pub name: Option<&'a str>,
    pub parameters: Option<&'a str>,
    pub return_type_clause: Option<&'a str>,
    pub body_clause: Option<&'a str>,
    // ... one field per declared AssembledField on the rule
}
```

**Contract**: For any corpus node, the TS `TemplateContext` and the Rust per-rule
struct receive byte-identical values in each field. Cross-render parity (FR-021)
depends on this.

## 2. Template asset layout

### Directory structure (per grammar)

```text
packages/rust/
├── src/               # generated TS sources (factories, types, etc.)
├── templates/         # [NEW] per-rule .jinja files
│   ├── function_item.jinja
│   ├── block.jinja
│   ├── struct_item.jinja
│   ├── visibility_modifier.jinja      # variant-branching (1 of 5)
│   ├── mod_item.jinja
│   ├── impl_item.jinja
│   └── ... (~160 files for Rust)
└── templates.yaml     # [DELETED in Phase A]
```

### File naming

- One `.jinja` file per rule.
- Filename is the rule's tree-sitter `kind` (snake_case), no prefix.
- Rule kinds follow tree-sitter's `[a-z_][a-z0-9_]*` convention — filesystem-safe on Linux, macOS, Windows.
- Hidden kinds (`_`-prefixed) that codegen emits templates for use the same convention (e.g., `_match_block.jinja`). The leading underscore is preserved — filesystems allow it.

### File header

Every generated `.jinja` file opens with a Jinja comment identifying it as generated:

```jinja
{# @generated by @sittir/codegen — do not edit. Source: packages/codegen/src/emitters/templates.ts #}
```

This satisfies Constitution Principle III (Generated vs Hand-Written).

### File contents

Content constraints (from research §3):

- Authoring subset only (interpolation, if/elif/else/endif, for with loop.\*, whitespace control, comments, 6 standardized filters).
- References only the variables defined in `TemplateContext`.
- No template inheritance, no macros, no `{% match %}`, no raw Rust/JS.

## 3. `Translator` (internal to codegen)

Lives in `packages/codegen/src/emitters/jinja-translator.ts`. Not part of the runtime or the published package surface — codegen-internal only.

### Signature

```ts
/**
 * Translate an assembled rule into a Jinja template body.
 *
 * Input is the canonical NodeMap entry (post-assemble, post-optimize).
 * Output is the text that `.jinja` file will contain (sans the
 * @generated header comment, which the emitter adds).
 *
 * Fails loudly (throws with rule name + unsupported construct) when the
 * rule uses a shape that cannot be represented in the Jinja subset.
 */
export function translateToJinja(
	node: AssembledNode,
	rules: Record<string, Rule>,
	wordMatcher: RegExp,
): string;
```

### Mapping rules (pure functions over `AssembledNode`)

See `contracts/translator-mapping.md` for the full mapping table. Rules are
implemented as a dispatch over `node.modelType`:

| modelType                             | Handler                                                                                                                                                      |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `branch`                              | Field + clause + children-slot substitution → single Jinja body                                                                                              |
| `container`                           | Mirrors `branch`, plus repeat-aware `{% for %}` / pre-joined children handling                                                                               |
| `polymorph`                           | If variant-branching (5 rules): emit `{% if variant == "<form>" %}` chain. Otherwise: emit the single-template form directly (collapsed identical variants). |
| `leaf`                                | `{{ text }}`                                                                                                                                                 |
| `keyword`, `token`                    | No template (renderer returns literal). Translator emits no file.                                                                                            |
| `enum`, `supertype`, `multi`, `group` | No template at the top level (resolved via child render). Translator emits no file unless a form needs one.                                                  |

### Failure modes (MUST throw, not degrade)

- Construct outside the authoring subset (e.g., a rule whose render needs recursion that Jinja can't express).
- Clause body that references an unnamed field or a field that doesn't appear on the node.
- Variant rule whose form names don't match the runtime `$variant` values (would silently fail dispatch).

## 4. Invariants

Must hold before and after every codegen regeneration:

1. **One template per rule**. For every rule whose `modelType` produces a template (branch / container / polymorph / leaf), exactly one `.jinja` file exists at `packages/<grammar>/templates/<kind>.jinja`.
2. **No stale files**. No `.jinja` file exists whose filename does not correspond to a rule in the grammar's `NodeMap`. Codegen emits a deterministic directory listing; files outside that set are deleted on regeneration.
3. **No YAML**. `packages/<grammar>/templates.yaml` does not exist post Phase A.
4. **Identical context shapes**. For every rule that has both a TS render path and a Rust render path, the Rust struct's field set is a superset of the TS `TemplateContext` keys used by that rule's `.jinja` file. askama build time enforces this (unused struct fields are fine; missing referenced fields are build errors).
5. **Byte-identical corpus output**. Per-phase round-trip corpus produces output bytes equal to the pre-migration baseline.

## 5. Relationships to existing entities

| Existing                                | New                      | Relationship                                                                                                                                                                                                                                                      |
| --------------------------------------- | ------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `NodeMap.nodes`                         | `.jinja` file set        | 1:(0 or 1) — every non-hidden, non-token-shaped `AssembledNode` gets one `.jinja` file.                                                                                                                                                                           |
| `PreparedRender` (from ADR-0013 Task 3) | `TemplateContext`        | Same bag, new formal contract. Phase A refactors the already-finished substitution loop into a `TemplateContext` → Nunjucks → string flow.                                                                                                                        |
| `RulesConfig` (from `@sittir/core`)     | Retired for rendering    | `RulesConfig.rules` no longer authoritative for templates. Retained for now as a lookup map for other metadata (joinBy, clauses); eventually replaced by per-rule metadata embedded in `.jinja` files via comments or sidecar, but out of scope for this feature. |
| `AssembledNode.renderTemplate`          | `translateToJinja(node)` | Retired. Moved from class method to external translator function — the class-method form coupled template emission to the class hierarchy in a way the new per-rule file model doesn't need.                                                                      |
